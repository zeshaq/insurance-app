Audit trail — slice 13

The slice that proves the difference between retention (Kafka keeps every record for N days) and compaction (Kafka keeps only the LATEST record per key, forever). Every claim state transition writes to BOTH the compacted audit-events topic (keyed by claim id) AND the retention-based claim-events topic. The contrast view below shows what each topic looks like for the same claim, side by side.

Pick a claim

Try this: file a fresh claim from the claim page, note the id, then come back and POST /api/claims/{id}/approve (via curl or DevTools — the GUI doesn't have an approve button yet). The contrast below will show: snapshot = APPROVED, events = [FILED, APPROVED].
