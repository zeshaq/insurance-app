Payment — slice 7

Two patterns paired in one slice: an Idempotency-Key header (client-supplied) makes every retry collapse onto the same charge, and a failing gateway triggers @Retry (3 attempts) then routes the record to a DLQ topic (payment-dlq) so operators can reconcile.

Pay against a policy

Use a policy number you got from the Policy page. Try amount = 100 for the happy path, then amount = 9999 for the DLQ demo (WireMock returns 503 on amounts ≥ 9000, exhausting @Retry).

Demo flows

Replay — leave the key the same, click submit twice. First time → 201 Created + a fresh externalRef. Second → 200 OK + the same externalRef. No double charge.
DLQ — amount 9999 → 502 Bad Gateway, status: FAILED in the body. Check Kafka UI → payment-dlq: a record keyed by your idempotency-key appears, carrying the failure reason + retry count.
